# Oracle stored-source filename extensions

Research date: 2026-07-16. Sources below are Oracle-owned documentation or official Oracle examples.

## Bottom line

Oracle Database does **not** impose filename extensions on stored source: the database compiles `CREATE ...` statements, while filenames belong to client tools and source-control layouts. Oracle's own Self-Service Automation documentation explicitly calls its extension mappings “guidelines only” and says they are not required.[1]

For Apex Pilot, the strongest Oracle-backed canonical mapping is:

- **Package specification:** `.pks`
- **Package body:** `.pkb`
- **Combined package specification + body:** `.sql` is directly generated by an Oracle tool; `.pck` is also used by Oracle product-delivered combined package files. No primary Oracle source found here establishes `.pkg` as the Oracle-wide combined-package convention.
- **Type specification / type body:** `.sql`
- **Standalone procedure / function / trigger:** `.sql`

## Evidence and confidence

- **`.pks` and `.pkb` are explicit Oracle defaults.** Oracle SQL\*Module says generated package source uses `.pks` for the specification and `.pkb` for the body.[2] Current SQLcl Projects exports supported objects under `src/database/<SCHEMA>/<OBJECT_TYPE>/<OBJECT_NAME>[.sql|.pks|.pkb]`; its supported object types include package spec/body, type spec/body, procedure, function, and trigger.[3]

- **`.sql` is Oracle-owned tooling's general stored-source format.** Oracle Developer Tools for Visual Studio requires managed project scripts to use `.sql` and can generate scripts for procedures, functions, triggers, and packages.[4] It explicitly writes a package's definitions to one `.sql` file,[5] writes object-type definitions and type bodies to `.sql`,[6][7] and writes triggers to `.sql`.[8] SQLcl's official project example exports standalone procedures and triggers as `.sql`.[9]

- **`.pck` is genuinely used by Oracle, but it is product/tool-specific rather than a database rule.** Oracle Developer Tools recognizes `.pck` in its SQL/PLSQL editor extension list.[10] Oracle Identity Governance ships files such as `OIM_EMPLOYEE_WRAPPER.pck` and calls them “package file[s]”; related instructions address both the package specification and body within such `.pck` wrapper packages, supporting `.pck` as a combined-package convention in that Oracle product family.[11][12]

- **`.pkg` is not verified as Oracle's combined-package convention.** Oracle E-Business Suite documentation contains a “PKG File Location Standard,” so Oracle-owned material does use the term/file type,[13] but the same standards say package specifications and bodies should be in different files. The reviewed current SQLcl and Oracle Developer Tools documentation does not list `.pkg`. Treat `.pkg = combined package` as an Apex Pilot/user convention unless a more specific Oracle tool contract requires it.

- **`.tps` / `.tpb` are not established by the reviewed Oracle primary sources.** Oracle's current source-management tools use `.sql` for type specifications and bodies.[3][6][7] These extensions may be useful compatibility aliases, but should not be labeled Oracle-defined based on this evidence.

- **`.prc`, `.fnc`, and `.trg` are Oracle-tool-recognized aliases, not current canonical export output.** Oracle Developer Tools lists all three among extensions its SQL/PLSQL editor can open,[10] but its generated project scripts use `.sql`, and SQLcl's official examples export procedures/triggers as `.sql`.[4][8][9]

## Current Apex Pilot frontend

- `frontend/src/editorLanguages.ts` maps `.sql`, `.pls`, `.plb`, `.pck`, `.pks`, `.pkb`, `.tps`, `.tpb`, `.fnc`, `.prc`, `.trg`, and `.vw` to Monaco SQL. It does **not** map `.pkg`.
- `frontend/src/fileTreeIcons.tsx` gives package icons to `pks`, `pkb`, `pls`, `plb`, `pck`, `spc`, `bdy`, `tps`, and `tpb`. It does **not** include `pkg`, `fnc`, `prc`, or `trg`.

This is broader compatibility handling than Oracle's documented canonical outputs. If the product wants to honor the confirmed user convention, `.pkg` must be added separately; retaining `.pck` is defensible because Oracle-owned tools and deliverables use it.
