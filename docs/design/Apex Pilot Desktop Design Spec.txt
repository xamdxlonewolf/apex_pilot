# Apex Pilot UI Design Specification
Version: 1.0 (Draft)
Status: Living Document

---

# 1. Purpose

This document defines the visual language, interaction patterns, layout system, and user experience standards for Apex Pilot.

It is the authoritative source for all desktop UI implementation.

Its purpose is to ensure every screen, component, workflow, and interaction follows a consistent design language while maintaining the core principles of Apex Pilot.

This specification intentionally focuses on user experience and interface implementation.

It does not define backend architecture, SQL execution logic, MCP implementation, Oracle integration, or AI prompt engineering.

---

# 2. Design Philosophy

Apex Pilot is not a dashboard.

It is not a web application.

It is not a SaaS product.

It is a professional desktop development environment.

The user should immediately recognize similarities to tools such as:

- JetBrains IDEs
- Visual Studio Code
- DataGrip
- SQL Developer
- DBeaver
- Postman

The application should feel like software intended to be used for eight hours a day.

Every decision in this specification supports that goal.

---

# 3. Core Principles

## 3.1 Developer First

The interface always prioritizes speed over decoration.

Additional clicks should never exist for aesthetic reasons.

Information should remain visible whenever practical.

Panels should expose functionality rather than hide it.

Power users should rarely need modal dialogs.

---

## 3.2 Trust Before Automation

AI never performs hidden work.

Every operation progresses through visible stages.

Intent

↓

Planning

↓

Generated SQL

↓

Review

↓

Execution

↓

Audit

The user should understand exactly what is about to happen before execution begins.

---

## 3.3 Context Never Disappears

The application should continuously communicate:

Current Project

Current Connection

Connected User

Working Schema

Current Environment

Backend Status

MCP Status

This information should never require navigation.

It belongs to the application shell.

---

## 3.4 Explainability

Every significant operation should answer:

What is happening?

Why is it happening?

What will execute?

Where will it execute?

How will it execute?

What happened?

The interface should never force users to infer important information.

---

## 3.5 Progressive Disclosure

Complexity should be revealed naturally.

Users should not be overwhelmed.

However...

Information should never be hidden behind unnecessary navigation.

Panels may collapse.

Information should not disappear.

---

## 3.6 Local First

Everything should reinforce that work occurs locally.

Projects are folders.

Connections belong to the user.

Profiles are local.

No cloud concepts appear anywhere within the interface.

There are no organizations.

No workspaces.

No tenants.

No team dashboards.

---

# 4. Overall User Experience

The emotional response to Apex Pilot should be:

"I trust this."

rather than

"This looks impressive."

Professional software earns trust through consistency.

Not animation.

Not gradients.

Not oversized cards.

The application should appear engineered.

Not marketed.

# 6. Information Density

The interface intentionally favors medium-to-high information density.

Large empty areas are discouraged.

Whitespace exists to improve readability.

Not aesthetics.

Users spend hours inside Apex Pilot.

Information density should increase productivity rather than create clutter.

# 8. Product Identity

Apex Pilot should communicate one central idea.

The AI is not the product.

The AI is the navigator.

SQLcl performs execution.

Oracle Database performs work.

Apex Pilot orchestrates the workflow.

This distinction should influence every interaction.

The interface should continually reinforce that the user remains in control.

# 9. Window Shell

## Purpose

The Window Shell is the permanent structure of Apex Pilot.

Unlike web applications, the shell is not replaced as the user navigates.

The shell establishes spatial memory.

Users should always know where to find:

- Project information
- Database connection
- AI conversation
- Database objects
- Execution status
- MCP activity

Changing projects or workflows should update panel contents rather than reorganize the application.

The shell should feel permanent.

---

# 26. Mission Workspace

## Purpose

The Mission Workspace is the primary interaction area of Apex Pilot.

Unlike traditional chat interfaces, it represents the lifecycle of work.

The user defines intent.

Apex Pilot plans the work.

The user reviews the plan.

SQL is generated.

SQL is reviewed.

Execution occurs.

Results are presented.

The Mission Workspace documents this process.

It is both a conversation and a permanent engineering record.

---

# 27. Design Philosophy

The Mission Workspace is NOT a messaging application.

Avoid visual patterns from:

- Slack
- Discord
- Teams
- Messenger

Instead draw inspiration from:

- GitHub Pull Requests
- GitHub Actions
- Cursor
- Claude Artifacts
- Build Pipelines
- CI/CD timelines

Messages are secondary.

Tasks are primary.

---

# 47. Inspector

## Purpose

The Inspector is the contextual intelligence panel of Apex Pilot.

It is responsible for explaining the current state of the active Mission.

The Inspector never initiates work.

The Inspector never owns execution.

The Inspector explains.

It provides visibility into:

- Current Mission
- Current Stage
- Generated SQL
- SQL Classification
- Objects Affected
- Dependencies
- Risks
- Execution Progress
- Results
- Audit Information

The Inspector is the primary trust mechanism of Apex Pilot.

---

# 48. Design Philosophy

Traditional AI applications require users to search conversations for important information.

Apex Pilot intentionally separates:

Intent

↓

Evidence

The Mission Workspace captures intent.

The Inspector presents evidence.

Users should never scroll through a conversation searching for SQL.

# 67. Project Explorer

## Purpose

The Project Explorer is the primary navigation panel for Apex Pilot.

It provides a unified view of:

- Local project files
- Oracle APEX application assets
- Database schema objects
- Working files
- Favorites
- Recent objects

Unlike traditional IDEs, the Project Explorer combines the local filesystem with live database metadata.

It is the primary navigation surface for the application.

It is never responsible for displaying object details.

Selection changes the active workspace.

Inspection occurs in the Inspector.

---

"What can I work on?"

It should never become cluttered with actions.

Navigation belongs here.

Actions belong in toolbars, context menus, and the Inspector.

The Explorer should remain visually quiet.

My Final Recommendation

After writing the equivalent of a full product specification, I think Apex Pilot has evolved into something much more distinctive than the original concept. If I were joining the project as the product design lead, I would make these recommendations before any significant implementation begins:

1. Lock the Information Architecture First

Treat these as the permanent top-level concepts of the application:

Explorer — What exists?
Mission — What am I trying to accomplish?
Workspace — What am I editing?
Inspector — Can I trust this?
Developer Console — What is the system doing?
Status Bar — What is the current environment and health?

Every future feature should fit naturally into one of these areas. If it doesn't, rethink the feature rather than adding another panel.

2. Make "Mission" the Defining Concept

This is the biggest opportunity to differentiate Apex Pilot.

Don't imitate chat-based AI tools.

Users should think in terms of missions, not conversations.

Every mission becomes:

A planning record
A review record
An execution record
An audit record
A reusable history item

That's a much stronger mental model for database development.

3. Make the Inspector the Trust Center

If users say, "I trust Apex Pilot," it won't be because of the AI.

It will be because the Inspector consistently answers:

What is going to happen?
Why is it happening?
Where is it happening?
What executed it?
What changed?
What can I do next?
4. Optimize for Long Sessions

Every design decision should assume users spend 8+ hours a day in Apex Pilot.

That means:

Dense layouts
Excellent keyboard support
Minimal modal dialogs
Consistent interaction patterns
Fast navigation
Low visual noise
5. Build a Real Design System

Before building features, implement the design system as reusable React components.

Create components for:

Panels
Accordions
Timeline
Information Blocks
Status Pills
Universal Object Cards
Inspector Sections
Mission Events
Console Tabs

If you build those first, the rest of the UI will come together much faster and remain visually consistent.
